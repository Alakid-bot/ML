# Profiling Dataset Schema

This document defines the CSV template for the Machine Learning Prediction part of the
ML-based Load Prediction for Network Services project.

The first version of the model uses supervised regression:

```text
resource allocation + current traffic + current KPI -> maximum supported load
```

The prediction target is `max_supported_load_mbps`.

## CSV File

Use this template file as the starting point:

```text
data/profiling_dataset_template.csv
```

When real measurements are available, copy it to:

```text
data/profiling_dataset.csv
```

## Required Columns

| Column | Role | Unit | Example | Description |
|---|---|---:|---:|---|
| `sample_id` | Metadata | - | `1` | Unique row identifier. |
| `service_name` | Metadata | - | `snort` | Name of the profiled network service. |
| `traffic_input_mbps` | Feature | Mbps | `200` | Offered input traffic/load generated by tools such as iPerf. |
| `cpu_cores` | Feature | cores | `2` | CPU cores allocated to the service/container. |
| `ram_gb` | Feature | GB | `4` | RAM allocated to the service/container. |
| `link_capacity_mbps` | Feature | Mbps | `1000` | Maximum link capacity available to the service. |
| `cpu_utilization_percent` | Feature | percent | `72.8` | Observed CPU utilization during the profiling run. |
| `memory_utilization_percent` | Feature | percent | `45.0` | Observed memory utilization during the profiling run. |
| `latency_ms` | Feature | ms | `20.3` | Observed average latency during the profiling run. |
| `throughput_mbps` | Feature | Mbps | `180.0` | Observed successful throughput during the profiling run. |
| `packet_loss_percent` | Feature | percent | `0.5` | Observed packet loss rate. Use `0` if there is no packet loss. |
| `max_supported_load_mbps` | Target | Mbps | `230` | Maximum load the service can support while staying within acceptable KPI limits. |

## Optional Columns

These columns are useful for analysis and traceability, but the first model does not need to
use them as input features.

| Column | Unit | Example | Description |
|---|---:|---:|---|
| `timestamp` | ISO 8601 | `2026-06-24T15:00:00Z` | Time when the sample was collected. |
| `latency_p95_ms` | ms | `35.2` | 95th percentile latency. Better than average latency for SLA analysis. |
| `run_duration_seconds` | seconds | `60` | Duration of the traffic generation run. |
| `kpi_status` | - | `pass` | Whether the run stayed within target KPI limits, for example `pass` or `fail`. |
| `notes` | - | `baseline run` | Free-form notes about the experiment. |

## Feature Columns for the First Model

The first training script should use these columns as model inputs:

```text
traffic_input_mbps
cpu_cores
ram_gb
link_capacity_mbps
cpu_utilization_percent
memory_utilization_percent
latency_ms
throughput_mbps
packet_loss_percent
```

The target column is:

```text
max_supported_load_mbps
```

## Data Collection Rules

1. Keep units consistent across all rows. Use Mbps, GB, milliseconds, and percentages.
2. Do not mix Mbps and Gbps in the same column.
3. Each row should describe one profiling run under one resource and traffic configuration.
4. Avoid empty cells in required columns.
5. Record enough different combinations of CPU, RAM, link capacity, and traffic input.
6. Start with a small test file of 20 to 50 rows, then expand toward 1000 or more rows.
7. If a KPI is unstable, record the average value over the profiling run.
8. If a run fails completely, keep it in a separate failure log unless the value of every required column is clear.

## Example Row Meaning

```csv
sample_id,service_name,traffic_input_mbps,cpu_cores,ram_gb,link_capacity_mbps,cpu_utilization_percent,memory_utilization_percent,latency_ms,throughput_mbps,packet_loss_percent,max_supported_load_mbps
1,snort,200,2,4,1000,72.8,45.0,20.3,180.0,0.5,230
```

This means:

```text
When the service `snort` receives 200 Mbps input traffic with 2 CPU cores, 4 GB RAM,
and 1000 Mbps link capacity, the observed KPI values are 72.8% CPU utilization,
45.0% memory utilization, 20.3 ms latency, 180 Mbps throughput, and 0.5% packet loss.
Under the defined acceptable KPI constraints, the estimated maximum supported load is 230 Mbps.
```
